# P0003 Base-load time-series simulator

## Purpose

Make the synthetic stock generated by P0002 deliver base-load power values for every 15-minute timestep across a full simulation period.

The package should move the base-load object contract from energy-first to power-first. Objects should primarily produce expected and actual kW for a timestep. Energy in kWh should be derived from kW and timestep duration.

## Business intent

The simulator needs a residual base-load time series because base load:

- consumes local production,
- uses battery energy,
- creates residual-load forecast uncertainty,
- reduces risk-adjusted bidable flexibility,
- affects customer and site energy economics.

P0003 should produce the first annual residual base-load profile for the synthetic stock. The initial target year is 2025.

## Scope

Implement:

```text
1. Power-first base-load object contract
2. 15-minute time index for 2025
3. Deterministic object-level ticking
4. Site, household and portfolio aggregation
5. Summary metrics for base-load level and volatility
```

## Power-first object contract

Base-load objects must expose kW methods as the primary interface:

```python
expected_kw(context) -> float
actual_kw(context, rng) -> float
```

Energy methods may remain for compatibility but must be derived from kW:

```python
expected_kwh(context) = expected_kw(context) * context.step_hours
actual_kwh(context, rng) = actual_kw(context, rng) * context.step_hours
```

All base-load objects are non-controllable load objects and must return non-negative kW.

## Simulation context

Extend or replace the minimal `SimulationContext` so it supports time-aware simulation.

Required fields:

```text
timestep

step_hours
```

Useful derived properties:

```text

weekday
hour
minute
quarter_of_day
```

Default `step_hours` is `0.25`.

## 2025 time index

Implement a helper that creates a 15-minute time index for 2025.

Required behaviour:

```text
start = 2025-01-01 00:00
end_exclusive = 2026-01-01 00:00
step_minutes = 15
```

Acceptance:

```text
365 days * 96 timesteps/day = 35040 timesteps
first timestamp = 2025-01-01 00:00
last timestamp = 2025-12-31 23:45
```

Use timezone-naive datetimes for the first implementation unless a later package introduces explicit timezone handling.

## Deterministic RNG

Object ticking must be reproducible and stable.

Do not use one global RNG whose output depends on iteration order. Use a deterministic per-object/per-timestep seed.

Suggested helper:

```python
stable_seed(base_seed, timestep, object_id) -> int
```

It should use a stable hash such as SHA-256, not Python's built-in `hash()`.

Then:

```python
rng = Random(stable_seed(config.seed, context.timestep, load_object.object_id))
```

This makes results reproducible even if later execution order changes.

Do not persist object-level contribution rows by default.

```text

```

## Performance expectations

P0003 must work for the P0002 default of 100 households over full-year 2025.

It should be written so later packages can scale to larger populations, but P0003 does not need to optimize for 10,000 households yet.

## Public-safe requirement

Use only synthetic and neutral examples. Do not add company names, real customer data, real prices, real production data or internal commercial terms.

## Acceptance criteria

### Object contract

- Every P0002 base-load object has `expected_kw(context)`.
- Every P0002 base-load object has `actual_kw(context, rng)`.
- Existing `expected_kwh(context)` is derived from `expected_kw(context) * context.step_hours`.
- Existing `actual_kwh(context, rng)` is derived from `actual_kw(context, rng) * context.step_hours`.
- All expected and actual kW values are non-negative.

### Time index

- 2025 simulation creates exactly `35040` timesteps.
- First timestamp is `2025-01-01 00:00`.
- Last timestamp is `2025-12-31 23:45`.
- `step_hours` is `0.25` for 15-minute simulation.

### Simulation

- Running the simulator for default P0002 stock produces site, household and portfolio time series.
- Site time series row count equals `site_count * 35040`.
- Household time series row count equals `household_count * 35040`.
- Portfolio time series row count equals `35040`.
- `actual_base_load_kwh = actual_base_load_kw * 0.25` at site and household level.
- `actual_base_load_mwh = actual_base_load_mw * 0.25` at portfolio level.
- Forecast error equals actual minus expected.

### Reproducibility

- Same stock seed and simulation seed produce identical time series.
- Deterministic per-object/per-timestep RNG does not depend on load-object iteration order for portfolio totals.

### Object behaviour

- Cold appliances vary over time in kW due to cycle logic.
- Internet and standby objects are near-constant with small noise.
- Always-light objects are usually on but not always on.
- Towel-rail objects are stochastic.

### Boundary

- Site, household and portfolio outputs expose aggregated base-load values only.
- Individual object states must not be included in normal simulator outputs.
- Object-level debug output, if implemented, must be clearly marked as debug/latent.

## Out of scope

- Behavioural residual loads such as cooking, shower, laundry or occupancy.
- Assets such as PV, battery, EV charger or controllable heat pump.
- Market bidding, baseline activation, rebound, settlement or business-case monetisation.
- Interactive web page.
- Calibration to real measurement data.
